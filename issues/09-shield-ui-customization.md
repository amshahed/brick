# Plan — Issue #9: Shield UI customization

## Context

Most of this issue's surface area is already in place from earlier slices:

- `ShieldConfigurationExtension.swift` returns a custom `ShieldConfiguration` with "Blocked by Brick" title, the shielded app's `localizedDisplayName` as subtitle, "Take a break" primary button, and a "Close" secondary button. `configuration(shielding:in:)` (category-level) already delegates to the app-level overload. Web-domain overloads return a default config (out of scope for MVP).
- `ShieldActionExtension.swift` handles `.primaryButtonPressed` by encoding the `ApplicationToken` and writing a `BreakIntent` plist to the App Group container, then returning `.defer`. `.secondaryButtonPressed` returns `.close`.
- `BreakIntent` (Shared/Services) already supports both the intent-file handoff *and* a `brick://break?token=<base64url>` URL entry point via `fromURL(_:)`.
- The main app's `Info.plist` (generated by `project.yml`) registers `brick://` under `CFBundleURLTypes`.
- `BrickApp.swift` wires `onOpenURL` → `intentInbox.handle(url:)` and `scenePhase == .active` → `intentInbox.checkForIntent()`. `HomeTab` observes `intentInbox.pending` and presents the break sheet with the token preselected.

The reason shield extensions can't literally "open the main app via URL" (as issue text imagines) is that `ShieldActionDelegate` can only return `.close` or `.defer` — Apple's API does not allow the extension process to open URLs. The intent-file + foreground-pickup pattern is the correct workaround and is already wired end-to-end.

## What's actually left

Small polish on the user-visible surface so the shield matches the spec's acceptance criteria and feels finished:

1. **Secondary button label**: spec says "OK", current code says "Close". Change to "OK".
2. **Primary button behavior**: currently returns `.defer`, which keeps the shield visible after the user taps "Take a break" — confusing because the button appears inert. Switch to `.close` so the shield dismisses; the user then opens Brick (dock/home/Spotlight) and `scenePhase == .active` picks up the intent. Still write the intent file before returning.
3. **Clear stale intents on "OK"**: if the user previously tapped "Take a break" (writing an intent) and then opened the shielded app again and tapped "OK", a stale intent could still be pending. Wipe any pending intent file on `.secondaryButtonPressed` so the next Brick launch doesn't unexpectedly pop the break sheet.
4. **Title/subtitle copy**: keep the two-line hierarchy but sharpen the subtitle to include a short, non-aggressive hint ("Tap 'Take a break' to unlock for a few minutes."). Memory-budget safe — just literal strings.
5. **Icon**: `ShieldConfiguration.icon` defaults to the shielded app's icon when `nil`, which is what we want — leave unset.

private func subtitleText(for app: Application) -> String {
    if let name = app.localizedDisplayName {
        return "\(name) is paused. Take a short break to unlock for a few minutes."
    }
    return "This app is paused. Take a short break to unlock for a few minutes."
}

## Edge cases

- **No `localizedDisplayName`.** Fallback copy used.
- **Category-level shielding.** Already delegates to the application-level configuration. Good.
- **Web-domain shielding.** Out of scope; returns default `ShieldConfiguration` unchanged.
- **User taps "Take a break" then never opens Brick.** Intent file is written with a 60-second freshness window; `BreakIntent.consume()` silently drops stale entries — no zombie state.
- **User taps "Take a break" repeatedly across multiple apps.** Each press overwrites `Intents/break.plist` with the latest `ApplicationToken`; the one that matters is the most recent, which is what the user sees on Home.

## Verification plan

1. `xcodegen generate && xcodebuild -scheme Brick -destination 'generic/platform=iOS' build` succeeds.
2. Existing tests still pass (no test changes expected; the extension files aren't covered by unit tests).
3. Manual (on-device, since shield extensions don't run in Simulator reliably):
   - Schedule Instagram for the current minute → wait for shield.
   - Shield shows "Blocked by Brick" + "Instagram is paused…" + "Take a break" / "OK".
   - Tap "OK" → shield dismisses, no intent queued, opening Brick lands on Home.
   - Re-open Instagram → shield re-appears. Tap "Take a break" → shield dismisses. Open Brick → break sheet appears with Instagram preselected.
   - Open Instagram, tap "Take a break", immediately re-open Instagram and tap "OK" → open Brick → no break sheet pops (stale intent cleared by OK).

## Out of scope

- Showing *why* the app is blocked (schedule name, one-shot reason) on the shield. Requires a tiny App Group context plist kept fresh by the monitor extension; defer unless UX demands it.
- Shield branding assets (custom logo via `ShieldConfiguration.icon`). The system-provided app icon is sufficient for MVP.
- Web-domain shielding UX (#MVP targets apps only).
- Lockdown-mode shield variants (#10 will add a different `ShieldConfiguration` path gated on a shared flag).
